## Overview

## Objectives

The project focuses on the following goals:

- Study the architecture, workflow, and core components of an Online Judge system.
- Build an Online Judge platform for automatic evaluation of `C`, `C++`, and `Python` submissions.
- Apply `Zero-shot Learning` to estimate whether source code is likely generated by AI.
- Integrate an alert mechanism to support instructors in academic integrity review.
- Evaluate the approach through experiments on human-written and AI-generated code samples.

## Repository scope

This repository currently focuses on two main areas:

- The Hydro-based Online Judge core for users, problems, submissions, and judging.
- A practical local development workflow that allows editing frontend and backend source code directly while still running the judge through Docker.

The AI-generated-code detection module is the thesis extension target and can be implemented as a dedicated analysis module or a separate service depending on the project stage.

## High-level architecture

- **Frontend**: user interface, mainly in `packages/ui-default`
- **Backend**: core web service and business logic in `packages/hydrooj`
- **Database**: MongoDB
- **Judge**: `hydrojudge` and sandbox running in Docker
- **AI Detection Module**: zero-shot-learning-based source code analysis module

## Recommended development workflow

The recommended setup for source-level development is:

- Frontend + Backend running locally from source
- Judge running separately in Docker

Do not use the old full Docker stack in `install/docker` if your main goal is to edit source code and see changes immediately.

### 1. Prerequisites

- Node.js 22
- Docker Desktop running Linux containers
- Local MongoDB

If you do not have local MongoDB yet, you can start one quickly with Docker:

```powershell
docker run -d --name hydro-mongo -p 27017:27017 mongo:7-jammy
```

### 2. Install dependencies

```powershell
cd C:\DATN\test\Hydro-master
corepack yarn install
```

### 3. Create local Hydro config

```powershell
New-Item -ItemType Directory -Force "$env:USERPROFILE\.hydro" | Out-Null
'{"url":"mongodb://localhost:27017/hydro"}' | Set-Content "$env:USERPROFILE\.hydro\config.json"
'["@hydrooj/ui-default"]' | Set-Content "$env:USERPROFILE\.hydro\addon.json"
```

### 4. Run frontend + backend locally

```powershell
cd C:\DATN\test\Hydro-master
corepack yarn dev:judge
```

This command will:

- run the backend on `0.0.0.0:2333`
- run the frontend dev server on `localhost:8000`
- watch source files and reload changes during development

### 5. Run the judge with Docker

Open another terminal:

```powershell
cd C:\DATN\test\Hydro-master\install\docker
docker compose -f docker-compose.local-judge.yml up -d --build
docker logs -f oj-judge-local
```

The Docker judge is configured to connect to the local backend through:

- `http://host.docker.internal:2333/`

### 6. Create the judge account for the first time

If the judge log shows `UserNotFoundError` or cannot find the `judge` account, create it with the CLI:

```powershell
cd C:\DATN\test\Hydro-master
node packages\hydrooj\bin\hydrooj.js cli user create systemjudge@systemjudge.local judge examplepassword auto
```

This command prints the new UID, for example `3`. Then grant judge privileges:

```powershell
node packages\hydrooj\bin\hydrooj.js cli user setJudge 3
```

If the returned UID is not `3`, replace `3` with the actual UID.

Then restart the Docker judge:

```powershell
cd C:\DATN\test\Hydro-master\install\docker
docker restart oj-judge-local
docker logs -f oj-judge-local
```

## Common issues

### Submission stays at `0 Waiting`

Typical reasons:

- Docker judge is not running
- local backend is not running
- backend is not listening on port `2333`

Quick checks:

```powershell
curl http://127.0.0.1:2333/status
docker logs -f oj-judge-local
```

### `connect ECONNREFUSED ...:2333`

The Docker judge cannot connect to the local backend.

Usually this means:

- `corepack yarn dev:judge` is not running
- the local backend has stopped
- port `2333` is not open

### `UserNotFoundError: User judge not found`

The local database does not contain the `judge` account yet. Create it using the CLI steps above.

### `No replica set found.`

This is only an informational MongoDB log in local development. It does not block judging.

### `sandbox version is vulnerable to symlink escape issue`

This is a sandbox warning from the judge image. It is not the reason the judge cannot connect to the backend.
